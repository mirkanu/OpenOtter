# Phase 01: Code Review Report

**Reviewed:** 2026-06-09T00:00:00Z
**Depth:** standard
**Files Reviewed:** 16

## Summary

Reviewed the OpenOtter audio processing core implementation including file upload, AssemblyAI transcription integration, and transcript display. The implementation follows the documented architecture but contains **2 critical defects** that must be fixed before shipping (broken file upload flow and data type mismatches), plus 4 warnings and 5 info-level issues.

The codebase is well-structured with proper TypeScript types, Zod validation, and client-side upload to bypass serverless limits. However, fundamental misunderstandings of Vercel Blob's client-side upload API and missing null safety checks will cause runtime failures.

## Critical Issues

### CR-01: Broken file upload flow - incorrect Vercel Blob usage

**File:** `src/lib/vercel-blob.ts:10-19`
**Issue:** `generateUploadToken()` is incorrectly implemented for client-side uploads. The function calls `put()` server-side with an empty `new Blob()`, which creates a zero-byte file, then returns the URL as a "token" for client-side upload. This breaks the entire upload flow because:

1. The server-side `put()` call creates an empty file at the URL
2. The client-side `upload()` expects a `handleUploadUrl` pointing to Vercel's upload API endpoint, not a blob URL
3. The returned `blob.url` is the final storage URL, not an upload URL

**Fix:**
```typescript
export async function generateUploadToken(filename: string): Promise<{ uploadUrl: string; token: string }> {
  // For client-side upload, we need to provide the BLOB_READ_WRITE_TOKEN
  const token = process.env.BLOB_READ_WRITE_TOKEN;
  
  if (!token) {
    throw new Error("BLOB_READ_WRITE_TOKEN environment variable is not set");
  }
  
  // Return the token and a placeholder URL
  // The actual URL will be generated by the client-side upload
  return { 
    uploadUrl: `https://blob.vercel-storage.com/${filename}`,
    token 
  };
}
```

The client-side `upload()` function from `@vercel/blob/client` handles the actual upload to Vercel's Blob storage using the token.

### CR-02: Runtime crash - missing null safety in transcript mapping

**File:** `src/lib/assemblyai.ts:59-65`
**Issue:** The `utterances?.map()` call will crash if `transcript.utterances` is `undefined`. TypeScript's optional chaining only protects the property access, not the method call on undefined. When AssemblyAI returns a response without utterances (e.g., during processing or error states), this will throw `TypeError: Cannot read property 'map' of undefined`.

**Fix:**
```typescript
export async function getTranscriptionResult(
  transcriptId: string
): Promise<TranscriptResult> {
  const client = getAssemblyAIClient();

  const transcript = await client.transcripts.get(transcriptId);

  return {
    id: transcript.id,
    status: transcript.status as 'processing' | 'completed' | 'error',
    text: transcript.text ?? undefined,
    utterances: transcript.utterances?.map(u => ({
      speaker: u.speaker,
      text: u.text,
      start: u.start,
      end: u.end,
      confidence: u.confidence,
    })) ?? [],
    error: transcript.error ?? undefined,
    audio_duration: transcript.audio_duration ?? undefined,
    confidence: transcript.confidence ?? undefined,
  };
}
```

## Warnings

### WR-01: Missing null safety in speaker color extraction

**File:** `src/components/TranscriptDisplay.tsx:29`
**Issue:** The regex `speaker.replace(/\D/g, "")` assumes speaker names always contain digits (e.g., "Speaker A" would return empty string). Calling `parseInt("")` returns `NaN`, and `colors[NaN % colors.length]` returns `undefined`, causing missing CSS classes and potential styling issues.

**Fix:**
```typescript
function getSpeakerColor(speaker: string): string {
  const colors = [
    "bg-indigo-100 text-indigo-800",
    "bg-emerald-100 text-emerald-800",
    "bg-amber-100 text-amber-800",
    "bg-rose-100 text-rose-800",
    "bg-cyan-100 text-cyan-800",
    "bg-violet-100 text-violet-800",
    "bg-pink-100 text-pink-800",
    "bg-teal-100 text-teal-800",
    "bg-orange-100 text-orange-800",
    "bg-lime-100 text-lime-800",
  ];
  // Extract number or use hash of string for non-numeric speakers
  const numericPart = speaker.replace(/\D/g, "");
  const index = numericPart 
    ? parseInt(numericPart) - 1 
    : speaker.split('').reduce((acc, char) => acc + char.charCodeAt(0), 0) % colors.length;
  
  return colors[Math.abs(index) % colors.length] || colors[0];
}
```

### WR-02: Unhandled promise rejection in polling cleanup

**File:** `src/app/transcript/[jobId]/page.tsx:52-54`
**Issue:** The polling interval continues firing even after an error occurs. If the API returns an error response, the component sets `loading=false` but the interval still runs, calling `pollTranscript()` which will repeatedly fail. This wastes API quota and creates console spam.

**Fix:**
```typescript
useEffect(() => {
  let intervalId: NodeJS.Timeout | null = null;
  let shouldContinue = true;

  const pollTranscript = async () => {
    if (!shouldContinue) return;
    
    try {
      const response = await fetch(`/api/transcribe/${jobId}`);
      if (!response.ok) {
        throw new Error("Failed to fetch transcript");
      }

      const data: TranscriptResult = await response.json();
      setTranscript(data);

      if (data.status === "completed" || data.status === "error") {
        setLoading(false);
        shouldContinue = false;
        if (intervalId) clearInterval(intervalId);
      }
    } catch (err) {
      if (shouldContinue) {
        const message = err instanceof Error ? err.message : "Failed to load transcript";
        setError(message);
        setLoading(false);
        shouldContinue = false;
        if (intervalId) clearInterval(intervalId);
      }
    }
  };

  // Initial poll
  pollTranscript();

  // Poll every 3 seconds if still processing
  if (loading) {
    intervalId = setInterval(() => {
      pollTranscript();
    }, 3000);
  }

  return () => {
    shouldContinue = false;
    if (intervalId) clearInterval(intervalId);
  };
}, [jobId]);
```

### WR-03: Missing file size validation - DoS risk

**File:** `src/components/AudioUploader.tsx:18-22`
**Issue:** No validation of file size before upload. Users can attempt to upload files larger than 5GB (Vercel Blob's limit) or extremely large files that will consume excessive quota. While Vercel Blob will reject oversized files, this wastes bandwidth and creates poor UX.

**Fix:**
```typescript
const handleFileSelect = useCallback(async (file: File) => {
  // Validate file type
  if (!isValidAudioFile(file)) {
    setError("Please select an MP3, WAV, or M4A audio file");
    return;
  }

  // Validate file size (5GB = 5 * 1024 * 1024 * 1024 bytes)
  const MAX_FILE_SIZE = 5 * 1024 * 1024 * 1024;
  if (file.size > MAX_FILE_SIZE) {
    setError("File size exceeds 5GB limit. Please use a smaller file.");
    return;
  }

  // Validate file is not empty
  if (file.size === 0) {
    setError("File is empty. Please select a valid audio file.");
    return;
  }

  setUploading(true);
  setProgress(0);
  setError(null);
  // ... rest of function
}, [onUploadComplete]);
```

### WR-04: Inconsistent error status handling

**File:** `src/lib/assemblyai.ts:57`
**Issue:** The status type assertion `as 'processing' | 'completed' | 'error'` is unsafe. AssemblyAI may return additional status values (e.g., "queued", "in_progress") that would be misclassified. The function could return `TranscriptResult` with an invalid status that breaks type safety throughout the app.

**Fix:**
```typescript
export async function getTranscriptionResult(
  transcriptId: string
): Promise<TranscriptResult> {
  const client = getAssemblyAIClient();

  const transcript = await client.transcripts.get(transcriptId);

  // Map AssemblyAI status to our simplified status
  let status: 'processing' | 'completed' | 'error';
  switch (transcript.status) {
    case 'completed':
      status = 'completed';
      break;
    case 'error':
    case 'cancelled':
      status = 'error';
      break;
    default:
      status = 'processing';
  }

  return {
    id: transcript.id,
    status,
    text: transcript.text ?? undefined,
    utterances: transcript.utterances?.map(u => ({
      speaker: u.speaker,
      text: u.text,
      start: u.start,
      end: u.end,
      confidence: u.confidence,
    })) ?? [],
    error: transcript.error ?? undefined,
    audio_duration: transcript.audio_duration ?? undefined,
    confidence: transcript.confidence ?? undefined,
  };
}
```

Update `src/lib/types.ts:16` to include additional AssemblyAI statuses if needed.

### IN-01: Unused contentType in UploadRequest

**File:** `src/lib/types.ts:31-34` and `src/app/api/upload/route.ts:10-13`
**Issue:** The `UploadRequest` interface includes `contentType` field, and the Zod schema marks it optional, but it's never used in the upload flow. The `AudioUploader` component only sends `filename`, and the `generateUploadToken` function ignores content type entirely. Remove to simplify interface.

**Fix:**
```typescript
// src/lib/types.ts
export interface UploadRequest {
  filename: string;
}

// src/app/api/upload/route.ts
const UploadRequestSchema = z.object({
  filename: z.string().min(1, "filename is required"),
});
```

### IN-02: Magic number in polling interval

**File:** `src/app/transcript/[jobId]/page.tsx:54`
**Issue:** Polling interval of 3000ms is hardcoded. AssemblyAI transcription time varies widely (30 seconds to several minutes), and a fixed 3-second interval may be inefficient for quick jobs or too slow for long ones.

**Fix:**
```typescript
// Consider exponential backoff or configuration
const POLL_INTERVALS = [2000, 3000, 5000, 8000, 10000]; // ms
let pollIndex = 0;

const getNextInterval = () => POLL_INTERVALS[Math.min(pollIndex++, POLL_INTERVALS.length - 1)];
```

### IN-03: Inconsistent error response status codes

**File:** `src/app/api/upload/route.ts:45`
**Issue:** Upload endpoint returns 503 (Service Unavailable) for all errors, but errors could be 400 (bad request), 401 (unauthorized), or 500 (internal error). Using 503 for everything makes debugging harder and misrepresents error types to clients.

**Fix:**
```typescript
return NextResponse.json(
  { error: message },
  { status: message.includes('environment variable') ? 500 : 503 }
);
```

### IN-04: Missing validation for AssemblyAI environment variable

**File:** `src/lib/assemblyai.ts:12-15`
**Issue:** The API key check throws an error during first client initialization, but this happens at runtime when the first API call is made, not during startup. A missing `ASSEMBLYAI_API_KEY` should be detected earlier to prevent partial failures.

**Fix:**
Add validation during module initialization:
```typescript
if (!process.env.ASSEMBLYAI_API_KEY) {
  throw new Error("ASSEMBLYAI_API_KEY environment variable is not set");
}

export function getAssemblyAIClient(): AssemblyAI {
  if (!client) {
    client = new AssemblyAI({ apiKey: process.env.ASSEMBLYAI_API_KEY! });
  }
  return client;
}
```

### IN-05: Hardcoded speaker count range

**File:** `src/lib/assemblyai.ts:35-36`
**Issue:** Speaker count range is hardcoded (2-10 speakers). This is arbitrary and may not suit all use cases. Some conversations have 1 speaker (monologues), others may have 10+ (panel discussions). This should be configurable or derived from user input.

**Fix:**
```typescript
export async function submitTranscription(
  audioUrl: string,
  options?: { minSpeakers?: number; maxSpeakers?: number }
): Promise<string> {
  const client = getAssemblyAIClient();

  const transcript = await client.transcripts.submit({
    audio: audioUrl,
    speaker_labels: true,
    speaker_options: {
      min_speakers_expected: options?.minSpeakers ?? 1,
      max_speakers_expected: options?.maxSpeakers ?? 10,
    },
  });

  return transcript.id;
}
```
