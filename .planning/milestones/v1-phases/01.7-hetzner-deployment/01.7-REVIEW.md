## Warnings

### WR-01: Error message silently discarded — user sees generic text, developer gets nothing

**File:** `src/app/page.tsx:14-16`

**Issue:** The `catch` block captures `err` into `error` but the rendered error UI on lines 20-23 shows only a hardcoded generic string. The actual `error` variable is never used in the returned JSX. When `getAllRecordings()` throws (e.g., DB path wrong, disk full, corrupt WAL file), the real error message is discarded entirely. There is no logging path either — the server-side error silently disappears. This makes production debugging nearly impossible.

**Fix:**
```tsx
// Option A: log server-side so it appears in PM2 logs
} catch (err) {
  error = err instanceof Error ? err.message : "Unknown error";
  console.error("[HomePage] Failed to load recordings:", err);
}

// Option B (if you also want it surfaced in dev):
if (error) {
  return (
    <div className="rounded-lg border border-destructive/50 bg-destructive/10 p-4 text-sm text-destructive">
      Could not load recordings. Refresh the page to try again.
      {process.env.NODE_ENV === 'development' && (
        <pre className="mt-2 text-xs">{error}</pre>
      )}
    </div>
  );
}
```

### WR-02: `as any[]` cast on raw DB rows defeats TypeScript type safety

**File:** `src/lib/recording-store.ts:56`

**Issue:** `.all() as any[]` bypasses TypeScript entirely. If a column is renamed or a migration adds a column with a different type, the cast silently succeeds and incorrect data flows through the manual field mapping on lines 58–72 without any compile-time signal. The same pattern is repeated in `getRecordingById` at line 83. Because `utterances_json` is parsed via `JSON.parse()` without schema validation (line 66, 95), a corrupt or malformed JSON string would throw an uncaught exception that propagates up through `getAllRecordings()` and, in `HomePage`, surfaces only as the generic error message (see WR-01 — both failures compound).

**Fix:**
```ts
// Replace unvalidated JSON.parse with safe parsing
utterances: row.utterances_json
  ? (() => {
      try { return JSON.parse(row.utterances_json as string); }
      catch { return undefined; }
    })()
  : undefined,
```

For the `as any[]` cast, either define a typed row interface matching the SELECT columns, or at minimum add a runtime check on `row.id` being a string before mapping.

### WR-03: `formatDate` silently renders "Invalid Date" when date strings are malformed

**File:** `src/lib/utils.ts:20-28` (called from `src/components/RecordingCard.tsx:17-19`)

**Issue:** `new Date(isoString)` is called with strings sourced from three different origins: SQLite `datetime('now')` (safe), client-supplied `file_created_at` (from `file.lastModified` in the browser, sent over HTTP), and audio metadata `recorded_at` (from music-metadata parsing). Malformed or non-ISO strings cause `new Date()` to return an `Invalid Date` object. `toLocaleDateString` on an `Invalid Date` returns the string `"Invalid Date"` — this renders visibly in the UI on `RecordingCard` with no indication something is wrong.

**Fix:**
```ts
export function formatDate(isoString: string | Date): string {
  const date = typeof isoString === 'string' ? new Date(isoString) : isoString;
  if (isNaN(date.getTime())) return 'Unknown date';
  return date.toLocaleDateString('en-US', {
    month: 'short',
    day: 'numeric',
    hour: '2-digit',
    minute: '2-digit',
  });
}
```

### IN-01: `formatDuration` returns `'--:--'` for valid zero-second durations

**File:** `src/lib/utils.ts:11`

**Issue:** The guard `if (!seconds)` is falsy for both `undefined` and `0`. A legitimate zero-second recording (e.g., a corrupt upload) would display `--:--` instead of `0:00`. This is a minor semantic inaccuracy.

**Fix:**
```ts
if (seconds == null || seconds === 0) return '--:--';
// or to show 0:00 for real zero durations:
if (seconds == null) return '--:--';
```

### IN-02: `RecordingMetadata.status` cast is unchecked at DB boundary

**File:** `src/lib/recording-store.ts:63`

**Issue:** `row.status as RecordingMetadata['status']` casts to the union `'queued' | 'processing' | 'completed' | 'error'` without validation. If a future code path writes an unexpected status string to the DB, the cast silently succeeds and the value flows through to `RecordingCard`'s `badgeVariant` switch — the badge would render the raw unknown string. Not urgent given single-user context, but worth a runtime check.

**Fix:**
```ts
const VALID_STATUSES = ['queued', 'processing', 'completed', 'error'] as const;
const status = VALID_STATUSES.includes(row.status)
  ? (row.status as RecordingMetadata['status'])
  : 'error'; // safe fallback
```
