# Phase 1.5: Recording Management - Research

**Researched:** 2026-06-10
**Domain:** Database persistence, audio playback, transcript synchronization
**Confidence:** HIGH

## Summary

Phase 1.5 implements the recording management layer that persists transcription results and enables users to browse, play, and read synchronized transcripts. The phase requires a database for recording metadata, audio player implementation with transcript synchronization, and two new pages (recordings list and individual recording view).

**Primary recommendation:** Use SQLite with `better-sqlite3` for simple, serverless persistence on Hetzner deployment. Implement HTML5 audio player with `timeupdate` event handling for transcript synchronization. Mobile-first UI with shadcn/ui components following project conventions.

## Architectural Responsibility Map

| Capability | Primary Tier | Secondary Tier | Rationale |
|------------|-------------|----------------|-----------|
| Recording metadata storage | API / Backend | — | Database CRUD operations happen server-side |
| Audio file serving | CDN / Static | API / Backend | Static audio files served directly from filesystem |
| Transcript synchronization | Browser / Client | API / Backend | Timestamp matching requires client-side audio API |
| Recording list display | Frontend Server (SSR) | Browser / Client | List pages benefit from SSR for performance |

**Data flow for recording playback:**
1. User clicks recording in list → navigation to `/recordings/[id]`
2. Page component fetches recording metadata from `/api/recordings/[id]`
3. Audio player loads file from `/api/audio/[filepath]` (streams from filesystem)
4. Transcript display renders utterances with timestamps
5. User plays audio → `timeupdate` events fire → transcript highlights current segment
6. User clicks transcript segment → audio player seeks to timestamp

### Pattern 1: Database Schema with better-sqlite3
**What:** Create SQLite database with recordings table for metadata persistence
**When to use:** Every transcription result must be saved for future access
**Example:**
```typescript
// Source: better-sqlite3 documentation
import Database from 'better-sqlite3';
import path from 'path';

const dbPath = path.join(process.env.HOME || '', 'openotter-recordings.db');
const db = new Database(dbPath);

// Create recordings table
db.exec(`
  CREATE TABLE IF NOT EXISTS recordings (
    id TEXT PRIMARY KEY,
    filename TEXT NOT NULL,
    filepath TEXT NOT NULL,
    status TEXT NOT NULL,
    audio_duration INTEGER,
    confidence REAL,
    speaker_count INTEGER,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
  )
`);

// Insert recording
const insertStmt = db.prepare(`
  INSERT INTO recordings (id, filename, filepath, status, audio_duration, confidence, speaker_count)
  VALUES (?, ?, ?, ?, ?, ?, ?)
`);
insertStmt.run(recordingId, filename, filepath, status, audioDuration, confidence, speakerCount);

// Query all recordings
const recordings = db.prepare('SELECT * FROM recordings ORDER BY created_at DESC').all();
```

### Pattern 2: HTML5 Audio Player with Transcript Sync
**What:** Synchronize transcript highlighting with audio playback using `timeupdate` events
**When to use:** Audio player needs to show which part of transcript is currently playing
**Example:**
```typescript
// Source: HTML5 Audio API documentation
'use client';

import { useEffect, useRef, useState } from 'react';

interface AudioPlayerProps {
  audioUrl: string;
  utterances: TranscriptUtterance[];
}

export function AudioPlayer({ audioUrl, utterances }: AudioPlayerProps) {
  const audioRef = useRef<HTMLAudioElement>(null);
  const [currentTime, setCurrentTime] = useState(0);

  const handleTimeUpdate = () => {
    if (audioRef.current) {
      setCurrentTime(audioRef.current.currentTime * 1000); // Convert to ms
    }
  };

  const currentUtterance = utterances.find(
    u => currentTime >= u.start && currentTime <= u.end
  );

  const handleClickUtterance = (startTime: number) => {
    if (audioRef.current) {
      audioRef.current.currentTime = startTime / 1000; // Convert to seconds
      audioRef.current.play();
    }
  };

  return (
    <div>
      <audio
        ref={audioRef}
        src={audioUrl}
        controls
        onTimeUpdate={handleTimeUpdate}
      />
      <div className="transcript">
        {utterances.map((u, i) => (
          <div
            key={i}
            className={currentUtterance === u ? 'highlight' : ''}
            onClick={() => handleClickUtterance(u.start)}
          >
            <strong>{u.speaker}:</strong> {u.text}
          </div>
        ))}
      </div>
    </div>
  );
}
```

### Anti-Patterns to Avoid
- **Loading all transcripts at once:** Only fetch recording metadata for list page. Load full transcript when user clicks individual recording.
- **Synchronous database operations in API routes:** better-sqlite3 is synchronous, which is fine for single-user, but use proper error handling.
- **Storing audio files in database:** Store only filepaths in database. Keep audio files on filesystem.
- **Complex audio player libraries:** HTML5 audio is sufficient. Don't add react-h5-audio-player or similar wrappers.
- **Forget to handle missing audio files:** Check if file exists before streaming. Return 404 if deleted.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Database queries | Raw SQL strings without prepared statements | better-sqlite3 prepared statements | Prevents SQL injection, handles escaping |
| Audio streaming | Custom buffer handling | Next.js `next()` or streaming response | Built-in streaming, proper headers |
| Form validation | Manual type checking | zod schemas (already installed) | Type-safe validation, consistent with existing code |
| ID generation | Random strings or timestamps | nanoid (already in dev dependencies) | Collision-resistant, URL-safe |
| Responsive UI | Custom CSS media queries | Tailwind CSS utility classes | Already configured, mobile-first by default |

**Key insight:** This phase uses basic web technologies that have standard solutions. Don't over-engineer audio players or database operations. SQLite and HTML5 audio are mature technologies with simple APIs.

## Common Pitfalls

### Pitfall 1: Audio File Path Security
**What goes wrong:** File traversal vulnerabilities expose files outside upload directory
**Why it happens:** Using user-provided filenames directly in filesystem paths
**How to avoid:**
- Store only sanitized filenames (replace non-alphanumeric chars with underscores)
- Validate filepath exists within upload directory before serving
- Use path.join and check resolved path is within UPLOAD_DIR
**Warning signs:** File API returns files with `../` or absolute paths

### Pitfall 2: Database Lock Contention
**What goes wrong:** Multiple writes fail with "database is locked" errors
**Why it happens:** better-sqlite3 uses synchronous writes; multiple concurrent operations can lock database
**How to avoid:**
- For single-user app, contention is unlikely (only one person uploading)
- Use WAL mode if needed: `db.pragma('journal_mode = WAL')`
- Keep transactions short
**Warning signs:** Intermittent database errors during concurrent operations

### Pitfall 3: Transcript Sync Performance
**What goes wrong:** Audio player lags when rendering long transcripts
**Why it happens:** Re-rendering entire transcript list on every `timeupdate` event (fires 4-10 times per second)
**How to avoid:**
- Use memoization: `useMemo` to find current utterance only when time changes significantly
- Throttle highlight updates or use CSS transitions
- Consider virtualization for very long transcripts (100+ utterances)
**Warning signs:** UI freezes during audio playback on mobile devices

### Pitfall 4: Missing File Cleanup
**What goes wrong:** Database references files that were deleted from filesystem
**Why it happens:** Files deleted manually or cleanup scripts don't update database
**How to avoid:**
- Always check file existence before streaming in API route
- Return 404 gracefully if file missing
- Optionally, add cleanup job to remove database entries for missing files
**Warning signs:** 404 errors when clicking play on old recordings

### Pitfall 5: Audio Format Compatibility
**What goes wrong:** Some audio files don't play in certain browsers
**Why it happens:** HTML5 audio support varies by format and browser
**How to avoid:**
- Stick to MP3, M4A, WAV (already validated in upload phase)
- Test on Safari iOS (project's target platform)
- Provide fallback message if format unsupported
**Warning signs:** Audio player shows error instead of playing

### Database Initialization and Schema
```typescript
// Source: better-sqlite3 documentation
import Database from 'better-sqlite3';
import path from 'path';
import fs from 'fs';

const DB_DIR = process.env.DB_DIR || path.join(process.env.HOME || '', 'openotter');
const DB_PATH = path.join(DB_DIR, 'recordings.db');

export function getDatabase(): Database.Database {
  // Ensure database directory exists
  if (!fs.existsSync(DB_DIR)) {
    fs.mkdirSync(DB_DIR, { recursive: true });
  }

  const db = new Database(DB_PATH);

  // Enable WAL mode for better concurrency
  db.pragma('journal_mode = WAL');

  // Create recordings table
  db.exec(`
    CREATE TABLE IF NOT EXISTS recordings (
      id TEXT PRIMARY KEY,
      filename TEXT NOT NULL,
      filepath TEXT NOT NULL,
      status TEXT NOT NULL,
      audio_duration INTEGER,
      confidence REAL,
      speaker_count INTEGER,
      utterances_json TEXT,
      created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
      updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
    )
  `);

  return db;
}
```

### Recording CRUD Operations
```typescript
// Source: better-sqlite3 prepared statements documentation
import { z } from 'zod';
import { nanoid } from 'nanoid';
import type { TranscriptResult } from './types';

const RecordingMetadataSchema = z.object({
  id: z.string().default(() => nanoid()),
  filename: z.string(),
  filepath: z.string(),
  status: z.enum(['queued', 'processing', 'completed', 'error']),
  audio_duration: z.number().optional(),
  confidence: z.number().optional(),
  speaker_count: z.number().optional(),
  utterances: z.array(z.object({
    speaker: z.string(),
    text: z.string(),
    start: z.number(),
    end: z.number(),
    confidence: z.number()
  })).optional(),
});

export function saveRecording(db: Database.Database, transcript: TranscriptResult, filename: string, filepath: string) {
  const speakerCount = transcript.utterances
    ? new Set(transcript.utterances.map(u => u.speaker)).size
    : 0;

  const stmt = db.prepare(`
    INSERT INTO recordings (id, filename, filepath, status, audio_duration, confidence, speaker_count, utterances_json)
    VALUES (?, ?, ?, ?, ?, ?, ?, ?)
  `);

  stmt.run(
    nanoid(),
    filename,
    filepath,
    transcript.status,
    transcript.audio_duration,
    transcript.confidence,
    speakerCount,
    transcript.utterances ? JSON.stringify(transcript.utterances) : null
  );
}

export function getAllRecordings(db: Database.Database) {
  const stmt = db.prepare(`
    SELECT id, filename, status, audio_duration, confidence, speaker_count, created_at
    FROM recordings
    ORDER BY created_at DESC
  `);

  return stmt.all();
}

export function getRecordingById(db: Database.Database, id: string) {
  const stmt = db.prepare(`
    SELECT * FROM recordings WHERE id = ?
  `);

  return stmt.get(id);
}
```

### Audio Streaming API Route
```typescript
// Source: Next.js API route documentation with streaming
import { NextRequest, NextResponse } from 'next/server';
import { createReadStream } from 'fs';
import { stat } from 'fs/promises';
import path from 'path';

const UPLOAD_DIR = process.env.UPLOAD_DIR || '/home/services/openotter/uploads';

export async function GET(
  request: NextRequest,
  { params }: { params: { filepath: string } }
) {
  const { filepath } = params;

  // Security: Validate filepath is within upload directory
  const fullPath = path.join(UPLOAD_DIR, filepath);
  const normalizedPath = path.normalize(fullPath);

  if (!normalizedPath.startsWith(UPLOAD_DIR)) {
    return NextResponse.json(
      { error: 'Invalid file path' },
      { status: 400 }
    );
  }

  try {
    const fileStats = await stat(normalizedPath);

    if (!fileStats.isFile()) {
      return NextResponse.json(
        { error: 'File not found' },
        { status: 404 }
      );
    }

    // Stream file
    const stream = createReadStream(normalizedPath);

    return new NextResponse(stream as any, {
      headers: {
        'Content-Type': 'audio/mpeg', // or detect from file extension
        'Content-Length': fileStats.size.toString(),
        'Cache-Control': 'public, max-age=31536000', // Cache audio files
      },
    });
  } catch (error) {
    return NextResponse.json(
      { error: 'File not found' },
      { status: 404 }
    );
  }
}
```

### Transcript Synchronization with Performance Optimization
```typescript
// Source: HTML5 Audio API + React optimization patterns
'use client';

import { useEffect, useRef, useState, useMemo } from 'react';
import type { TranscriptUtterance } from '@/lib/types';

interface SyncedTranscriptProps {
  utterances: TranscriptUtterance[];
}

export function SyncedTranscript({ utterances }: SyncedTranscriptProps) {
  const audioRef = useRef<HTMLAudioElement>(null);
  const [currentTime, setCurrentTime] = useState(0);
  const [isPlaying, setIsPlaying] = useState(false);

  // Throttle updates to avoid excessive re-renders
  const lastUpdateTime = useRef<number>(0);

  const handleTimeUpdate = () => {
    if (!audioRef.current) return;

    const now = Date.now();
    // Only update every 100ms
    if (now - lastUpdateTime.current < 100) return;

    lastUpdateTime.current = now;
    setCurrentTime(audioRef.current.currentTime * 1000);
  };

  // Memoize current utterance calculation
  const currentUtterance = useMemo(() => {
    return utterances.find(
      u => currentTime >= u.start && currentTime <= u.end
    );
  }, [currentTime, utterances]);

  const handleClickSegment = (startTime: number) => {
    if (!audioRef.current) return;

    audioRef.current.currentTime = startTime / 1000;
    audioRef.current.play();
    setIsPlaying(true);
  };

  return (
    <div className="space-y-4">
      <audio
        ref={audioRef}
        src="/api/audio/recording.m4a"
        controls
        onTimeUpdate={handleTimeUpdate}
        onPlay={() => setIsPlaying(true)}
        onPause={() => setIsPlaying(false)}
      />

      <div className="transcript space-y-3">
        {utterances.map((utterance, index) => {
          const isActive = currentUtterance === utterance;

          return (
            <div
              key={index}
              className={`p-3 rounded cursor-pointer transition-colors ${
                isActive
                  ? 'bg-blue-100 border-l-4 border-blue-500'
                  : 'bg-gray-50 hover:bg-gray-100'
              }`}
              onClick={() => handleClickSegment(utterance.start)}
            >
              <div className="font-semibold text-sm text-gray-700">
                {utterance.speaker}
              </div>
              <div className="text-gray-900 mt-1">
                {utterance.text}
              </div>
              <div className="text-xs text-gray-500 mt-1">
                {Math.round(utterance.start / 1000)}s - {Math.round(utterance.end / 1000)}s
              </div>
            </div>
          );
        })}
      </div>
    </div>
  );
}
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| SQL injection via string concatenation | Prepared statements in better-sqlite3 | Always been standard | Prepared statements are the only safe way to build SQL queries |
| Synchronous file reads in API routes | Streaming responses with createReadStream | Node.js stream API mature | Streaming reduces memory usage for large audio files |
| Re-rendering entire list on every tick | Memoized current utterance calculation | React hooks introduced | useMemo prevents unnecessary re-renders during audio playback |

**Deprecated/outdated:**
- **String concatenation for SQL queries:** Never use. Always use prepared statements or parameterized queries.
- **Loading entire file into memory before sending:** Outdated. Use streams for any file larger than a few MB.
- **jQuery-style DOM manipulation:** Modern React uses declarative components. Don't manipulate DOM directly for audio sync.

## Open Questions

1. **Question:** Should recordings include the full transcript JSON in the database, or store a reference to AssemblyAI transcript ID?
   - What we know: Both approaches work. Storing JSON duplicates data but makes queries self-contained. Storing reference saves space.
   - What's unclear: User's preference for data duplication vs. API dependency.
   - Recommendation: Store utterances JSON in database. This makes the app self-contained and not dependent on AssemblyAI API availability for historical recordings.

2. **Question:** Should recordings be deletable by the user?
   - What we know: Not explicitly in requirements. Storage cleanup is important for long-running app.
   - What's unclear: Whether delete functionality is in scope for Phase 1.5.
   - Recommendation: Include delete button on individual recording view. Add confirmation dialog. Delete both database entry and audio file.

3. **Question:** Should the audio player support variable playback speed?
   - What we know: HTML5 audio supports `playbackRate` property. Users often want 1.5x or 2x speed for long conversations.
   - What's unclear: Whether this is required for MVP or can be deferred.
   - Recommendation: Add speed controls (1x, 1.25x, 1.5x, 2x) using HTML5 audio API. Simple to implement and highly valuable for users.
