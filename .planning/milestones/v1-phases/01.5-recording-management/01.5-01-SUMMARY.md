## What Was Built

### `src/lib/database.ts`
Singleton `getDatabase()` function using better-sqlite3. Opens the SQLite database at `DATABASE_PATH` environment variable (default `/home/services/openotter/openotter.db`). Enables WAL journal mode for concurrent reads. Creates the `recordings` table if it does not exist with columns: `id`, `filename`, `filepath` (local path), `status`, `audio_duration`, `confidence`, `speaker_count`, `utterances_json`, `created_at`, `updated_at`.

### `src/lib/recording-store.ts`
Four synchronous CRUD functions (no async/await — better-sqlite3 design):
- `saveRecording(transcript, filename, filepath)` — inserts with nanoid, computes `speaker_count` via Set deduplication, serializes utterances as JSON, returns generated id
- `getAllRecordings()` — returns `RecordingMetadata[]` ordered by `created_at DESC`, parses utterances JSON
- `getRecordingById(id)` — returns `RecordingMetadata | null`, parses utterances JSON
- `deleteRecording(id)` — removes by id, returns `boolean` via `info.changes > 0`

All queries use `db.prepare(...).run()/.get()/.all()` with `?` placeholders (SQL injection protection, per threat T-1.5-01).

### `src/lib/types.ts` — `RecordingMetadata` interface added
```typescript
export interface RecordingMetadata {
  id: string;
  filename: string;
  filepath: string; // local filesystem path
  status: 'queued' | 'processing' | 'completed' | 'error';
  audio_duration?: number;
  confidence?: number;
  speaker_count?: number;
  utterances?: TranscriptUtterance[];
  created_at: string; // SQLite datetime string (TEXT column)
  updated_at: string; // SQLite datetime string (TEXT column)
}
```

## Decisions Made

1. **better-sqlite3 synchronous design** — All CRUD operations are synchronous. better-sqlite3 is designed to be synchronous; using it with async/await would add unnecessary complexity.
2. **utterances as TEXT/JSON** — Storing the full utterances array as a JSON string in a single `utterances_json` TEXT column avoids creating a separate `utterances` table for a single-user app.
3. **WAL mode** — Enables concurrent reads while maintaining single-writer semantics; appropriate for a personal-use app with occasional reads during writes.
