Five API endpoints providing the HTTP interface to the recording database layer, including a local-file audio proxy that prevents path traversal by resolving filepaths from DB records.

## What Was Built

**POST /api/recordings** ‚Äî Accepts `{ transcript, filename, filepath }`, validates with Zod schema (filepath as `z.string().min(1)` ‚Äî local path, not URL), calls `saveRecording` (synchronous), returns 201 with recording ID.

**GET /api/recordings** ‚Äî Returns `{ recordings: RecordingMetadata[] }` ordered by `created_at DESC` via synchronous `getAllRecordings()`.

**GET /api/recordings/[id]** ‚Äî Returns individual recording or 404. Uses Next.js 15 async params pattern.

**DELETE /api/recordings/[id]** ‚Äî Removes DB entry then deletes local audio file via `deleteFile()` (best-effort ‚Äî continues if file missing). Returns 404 if recording not found.

**GET /api/audio/[id]** ‚Äî Audio proxy: looks up recording by ID in DB, reads `recording.filepath` (trusted DB value ‚Äî never user input), streams with correct `Content-Type`, `Accept-Ranges`, `Content-Length`, `Cache-Control` headers. ENOENT returns 404.

**POST /api/transcribe** (updated) ‚Äî Now accepts `{ filepath, filename }` (local path) instead of `{ audioUrl }` URL, aligning with local filesystem migration. Returns `filename` and `filepath` in response for client to pass back during polling.

**GET /api/transcribe/[jobId]** (updated) ‚Äî Calls `saveRecording` (synchronous) when status is `completed` or `error`, using `filename` and `filepath` query params supplied by the client.
