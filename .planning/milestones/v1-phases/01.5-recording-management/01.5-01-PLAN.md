<objective>
Establish the database persistence layer for recording metadata using SQLite (better-sqlite3). This plan creates the database singleton initialization (WAL mode), synchronous recording CRUD operations, and type definitions needed to persist transcription results and retrieve them for the recordings list and detail views.

Purpose: Enable recording persistence with a zero-config embedded database. better-sqlite3 is synchronous, requires no connection pooling, and is perfectly suited for a single-user Hetzner VPS deployment. The database file path comes from the DATABASE_PATH env var (default `/home/services/openotter/openotter.db`). better-sqlite3 uses prepared statements with `?` placeholders for SQL-injection protection.

    ```typescript
    import Database from 'better-sqlite3';

    let db: Database.Database | null = null;

    // Returns a singleton better-sqlite3 Database instance.
    // Opens the database at DATABASE_PATH (default: /home/services/openotter/openotter.db).
    // Sets WAL journal mode for concurrent reads.
    // Creates the recordings table if it does not exist.
    // filepath column stores a LOCAL filesystem path (not a URL).
    export function getDatabase(): Database.Database {
      if (db) return db;

      const dbPath = process.env.DATABASE_PATH || '/home/services/openotter/openotter.db';
      db = new Database(dbPath);

      // WAL mode: better performance for concurrent reads, single-user write
      db.pragma('journal_mode = WAL');

      db.exec(`
        CREATE TABLE IF NOT EXISTS recordings (
          id TEXT PRIMARY KEY,
          filename TEXT NOT NULL,
          filepath TEXT NOT NULL,
          status TEXT NOT NULL,
          audio_duration INTEGER,
          confidence REAL,
          speaker_count INTEGER,
          utterances_json TEXT,
          created_at TEXT DEFAULT (datetime('now')),
          updated_at TEXT DEFAULT (datetime('now'))
        )
      `);

      return db;
    }
    ```
    
    Key implementation details:
    - Singleton pattern: create once per process, reuse on subsequent calls
    - DATABASE_PATH env var (default /home/services/openotter/openotter.db)
    - PRAGMA journal_mode = WAL (better concurrency for reads)
    - CREATE TABLE IF NOT EXISTS (safe to call multiple times)
    - filepath column stores LOCAL filesystem path (not a URL)
    - utterances_json stores full transcript as JSON string
    - created_at/updated_at stored as TEXT (SQLite datetime strings)
    - better-sqlite3 is fully synchronous — no async/await

    Create src/lib/recording-store.ts with synchronous CRUD operations using better-sqlite3:
    
    ```typescript
    import { getDatabase } from './database';
    import { nanoid } from 'nanoid';
    import type { TranscriptResult, TranscriptUtterance } from './types';
    import type { RecordingMetadata } from './types';

    export function saveRecording(
      transcript: TranscriptResult,
      filename: string,
      filepath: string  // local filesystem path
    ): string {
      const db = getDatabase();

      const speakerCount = transcript.utterances
        ? new Set(transcript.utterances.map(u => u.speaker)).size
        : 0;

      const recordingId = nanoid();

      const stmt = db.prepare(`
        INSERT INTO recordings (id, filename, filepath, status, audio_duration, confidence, speaker_count, utterances_json)
        VALUES (?, ?, ?, ?, ?, ?, ?, ?)
      `);

      stmt.run(
        recordingId,
        filename,
        filepath,
        transcript.status,
        transcript.audio_duration ?? null,
        transcript.confidence ?? null,
        speakerCount,
        transcript.utterances ? JSON.stringify(transcript.utterances) : null
      );

      return recordingId;
    }

    export function getAllRecordings(): RecordingMetadata[] {
      const db = getDatabase();

      const rows = db.prepare(`
        SELECT id, filename, filepath, status, audio_duration, confidence, speaker_count, utterances_json, created_at, updated_at
        FROM recordings
        ORDER BY created_at DESC
      `).all() as any[];

      return rows.map(row => ({
        id: row.id as string,
        filename: row.filename as string,
        filepath: row.filepath as string,  // local filesystem path
        status: row.status as RecordingMetadata['status'],
        audio_duration: row.audio_duration as number | undefined,
        confidence: row.confidence as number | undefined,
        speaker_count: row.speaker_count as number | undefined,
        utterances: row.utterances_json ? JSON.parse(row.utterances_json as string) : undefined,
        created_at: row.created_at as string,
        updated_at: row.updated_at as string,
      }));
    }

    export function getRecordingById(id: string): RecordingMetadata | null {
      const db = getDatabase();

      const row = db.prepare(`
        SELECT id, filename, filepath, status, audio_duration, confidence, speaker_count, utterances_json, created_at, updated_at
        FROM recordings
        WHERE id = ?
      `).get(id) as any;

      if (!row) return null;

      return {
        id: row.id as string,
        filename: row.filename as string,
        filepath: row.filepath as string,  // local filesystem path
        status: row.status as RecordingMetadata['status'],
        audio_duration: row.audio_duration as number | undefined,
        confidence: row.confidence as number | undefined,
        speaker_count: row.speaker_count as number | undefined,
        utterances: row.utterances_json ? JSON.parse(row.utterances_json as string) : undefined,
        created_at: row.created_at as string,
        updated_at: row.updated_at as string,
      };
    }

    export function deleteRecording(id: string): boolean {
      const db = getDatabase();
      const info = db.prepare('DELETE FROM recordings WHERE id = ?').run(id);
      return info.changes > 0;
    }
    ```
    
    Key implementation details:
    - ALL functions are SYNCHRONOUS (no async/await) — better-sqlite3 is synchronous
    - Use `db.prepare(...).run(...)` / `.get(...)` / `.all(...)` with `?` placeholders (prepared statements — SQL-injection safe)
    - saveRecording: uses nanoid, computes speaker_count from a Set, JSON.stringify utterances, returns id string
    - getAllRecordings: returns RecordingMetadata[] ordered by created_at DESC, JSON.parse utterances_json
    - getRecordingById: returns RecordingMetadata | null (null if not found)
    - deleteRecording: returns boolean (`info.changes > 0`)
    - filepath is a LOCAL filesystem path throughout (not a URL)
    - created_at/updated_at are TEXT strings (SQLite datetime format)

| T-1.5-01 | Injection (SQLi) | recording-store.ts CRUD operations | mitigate | Use better-sqlite3 prepared statements with `?` placeholders; never concatenate user input into SQL. db.prepare('...').run(val) automatically parameterizes all values. |

| T-1.5-03 | Information disclosure | Database file | accept | Database file resides on the Hetzner VPS filesystem at DATABASE_PATH, protected by OS permissions; no credentials in code. |
| T-1.5-04 | Denial of Service | Concurrent writes | accept | SQLite is a single-file embedded DB; WAL mode handles concurrent reads. Single-user app — no pooling concern. |
