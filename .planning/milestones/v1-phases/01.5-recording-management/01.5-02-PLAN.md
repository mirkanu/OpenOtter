<objective>
Create API routes for recording CRUD operations and integrate with the existing transcription flow. These routes enable the frontend to list recordings, fetch individual recording details, save transcription results, and delete recordings. Because audio files live on the server filesystem (local paths), the browser cannot use them as an `<audio src>` directly — so this plan also adds a GET /api/audio/[id] proxy that looks up the recording by ID, reads the file at `recording.filepath`, and streams it with the correct content-type. The transcription endpoint is updated to automatically save completed transcriptions to the database.

Purpose: Provide HTTP interface to recording database layer. Frontend components (recording list, detail view) consume these routes. Transcription flow automatically saves results to database. Audio proxy serves local files to the browser.

Output: Five API endpoints (POST/GET /api/recordings, GET/DELETE /api/recordings/[id], GET /api/audio/[id], updated POST /api/transcribe) with proper validation, error handling, and security controls.
</objective>

    import { NextRequest, NextResponse } from "next/server";
    import { z } from "zod";
    import { saveRecording, getAllRecordings } from "@/lib/recording-store";
    import type { TranscriptResult } from "@/lib/types";

    export const dynamic = "force-dynamic";

    // Zod schema for POST request validation
    const SaveRecordingSchema = z.object({
      transcript: z.object({
        id: z.string(),
        status: z.enum(['processing', 'completed', 'error']),
        text: z.string().optional(),
        utterances: z.array(z.object({
          speaker: z.string(),
          text: z.string(),
          start: z.number(),
          end: z.number(),
          confidence: z.number(),
        })).optional(),
        error: z.string().optional(),
        audio_duration: z.number().optional(),
        confidence: z.number().optional(),
      }),
      filename: z.string().min(1),
      filepath: z.string().min(1), // local filesystem path
    });

    // POST: Save transcription result to database
    export async function POST(request: NextRequest) {
      try {
        const body = await request.json();

        // Validate request body
        const validation = SaveRecordingSchema.safeParse(body);
        if (!validation.success) {
          return NextResponse.json(
            { error: validation.error.issues[0].message },
            { status: 400 }
          );
        }

        const { transcript, filename, filepath } = validation.data;

        // Save recording to database (synchronous better-sqlite3 call)
        const recordingId = saveRecording(transcript as TranscriptResult, filename, filepath);

        return NextResponse.json(
          { id: recordingId, message: "Recording saved" },
          { status: 201 }
        );
      } catch (error) {
        const message = error instanceof Error ? error.message : "Unknown error";
        console.error("Failed to save recording:", error);

        return NextResponse.json(
          { error: message },
          { status: 500 }
        );
      }
    }

    // GET: List all recordings
    export async function GET() {
      try {
        // Synchronous better-sqlite3 call
        const recordings = getAllRecordings();

        return NextResponse.json({ recordings }, { status: 200 });
      } catch (error) {
        const message = error instanceof Error ? error.message : "Unknown error";
        console.error("Failed to fetch recordings:", error);

        return NextResponse.json(
          { error: message },
          { status: 500 }
        );
      }
    }
    ```
    
    Key implementation details:
    - POST validates transcript structure using Zod schema matching TranscriptResult
    - POST validates filepath as a non-empty string (local filesystem path — NOT z.string().url())
    - POST returns 201 with recording ID on success
    - GET returns { recordings: RecordingMetadata[] } wrapper
    - Both handlers use try/catch with proper error logging
    - Force dynamic to prevent caching
    - saveRecording and getAllRecordings are SYNCHRONOUS (better-sqlite3) — no await needed
  </action>

      } catch (error) {

        console.error("Failed to fetch recording:", error);

        return NextResponse.json(
          { error: message },
          { status: 500 }
        );
      }

    // DELETE: Remove recording and associated local audio file
    export async function DELETE(
      request: NextRequest,
      { params }: { params: Promise<{ id: string }> }
    ) {
      try {
        const { id } = await params;

        if (!id) {
          return NextResponse.json(
            { error: "Recording ID is required" },
            { status: 400 }
          );
        }

        // Synchronous better-sqlite3 call
        const recording = getRecordingById(id);

        if (!recording) {
          return NextResponse.json(
            { error: "Recording not found" },
            { status: 404 }
          );
        }

        // Delete recording from database (synchronous)
        const deleted = deleteRecording(id);

        if (deleted && recording.filepath) {
          // Delete local audio file — best-effort (continue even if file missing)
          try {
            await deleteFile(recording.filepath);
          } catch (fileError) {
            console.warn(`Failed to delete local file ${recording.filepath}:`, fileError);
            // Continue even if file deletion fails — database entry is removed
          }
        }

        return NextResponse.json(
          { message: "Recording deleted" },
          { status: 200 }
        );
      } catch (error) {
        const message = error instanceof Error ? error.message : "Unknown error";
        console.error("Failed to delete recording:", error);

        return NextResponse.json(
          { error: message },
          { status: 500 }
        );
      }
    }
    ```

    - DELETE calls deleteFile from vercel-blob.ts for local file cleanup (filepath is a local path)

    Create src/app/api/audio/[id]/route.ts — a proxy that reads a local audio file by recording ID and streams it to the browser:
    
    Note: This proxy is REQUIRED because local filesystem paths cannot be used as a browser audio `src`. The [id] URL segment is only used as a database lookup key — the actual filepath comes from `recording.filepath` (a trusted DB value), never from the URL.
    
    ```typescript
    import { NextRequest, NextResponse } from "next/server";
    import { readFile } from "fs/promises";
    import path from "path";
    import { getRecordingById } from "@/lib/recording-store";

    export const dynamic = "force-dynamic";

    function getContentType(filepath: string): string {
      const ext = path.extname(filepath).toLowerCase();
      switch (ext) {
        case '.m4a': return 'audio/mp4';
        case '.mp3': return 'audio/mpeg';
        case '.wav': return 'audio/wav';
        case '.ogg': return 'audio/ogg';
        case '.webm': return 'audio/webm';
        default: return 'application/octet-stream';
      }
    }

    export async function GET(
      request: NextRequest,
      { params }: { params: Promise<{ id: string }> }
    ) {
      try {
        const { id } = await params;

        if (!id) {
          return NextResponse.json(
            { error: "Recording ID is required" },
            { status: 400 }
          );
        }

        // Synchronous DB lookup — filepath comes from the trusted DB record, not the URL
        const recording = getRecordingById(id);

        if (!recording) {
          return NextResponse.json(
            { error: "Recording not found" },
            { status: 404 }
          );
        }

        // Read local audio file
        const fileBuffer = await readFile(recording.filepath);
        const contentType = getContentType(recording.filepath);

        return new NextResponse(fileBuffer, {
          status: 200,
          headers: {
            'Content-Type': contentType,
            'Accept-Ranges': 'bytes',
            'Content-Length': String(fileBuffer.length),
            'Cache-Control': 'private, max-age=3600',
          },
        });
      } catch (error) {
        const message = error instanceof Error ? error.message : "Unknown error";
        console.error("Failed to stream audio file:", error);

        // Return 404 if file not found on disk
        if (error instanceof Error && 'code' in error && (error as NodeJS.ErrnoException).code === 'ENOENT') {
          return NextResponse.json(
            { error: "Audio file not found" },
            { status: 404 }
          );
        }

        return NextResponse.json(
          { error: message },
          { status: 500 }
        );
      }
    }
    ```
    
    Key implementation details:
    - The [id] param is ONLY used as a DB lookup key — never constructed into a file path
    - recording.filepath (from DB) is the trusted source of the actual file path
    - Content-type inferred from file extension (.m4a → audio/mp4, etc.)
    - Accept-Ranges: bytes header enables browser seeking
    - ENOENT error → 404 (file deleted from disk but still in DB)
    - getRecordingById is SYNCHRONOUS (better-sqlite3) — no await
  </action>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → API → Database | User-provided recording IDs cross into database queries |
| Client → API → local filesystem | Recording IDs trigger local file reads and deletions |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1.5-05 | Injection (SQLi) | /api/recordings routes | mitigate | Use better-sqlite3 prepared statements via recording-store.ts (inherited from Plan 01). No raw SQL in API routes. |
| T-1.5-06 | Tampering | Recording ID in URL | accept | Recording ID validated by database query. Invalid IDs return 404. No user-provided IDs in SQL directly. |
| T-1.5-07 | Information Disclosure | Local file paths in responses | accept | Files are behind Cloudflare Access. filepath is a server-side path; never expose it in API responses to the client. |
| T-1.5-08 | Tampering | Filename/filepath from client | mitigate | Validate filepath as a non-empty string in POST /api/recordings. Only save recordings from trusted AssemblyAI flow. |
| T-1.5-09 | Denial of Service | Large transcription saves | accept | Single-user app makes DoS unlikely. Acceptable for MVP. |
| T-1.5-PROXY | Injection (Path traversal) | /api/audio/[id] proxy | mitigate | File path is read from the trusted DB record (recording.filepath), never constructed from the user-supplied URL segment. The [id] param is only used as a DB lookup key. |
</threat_model>
