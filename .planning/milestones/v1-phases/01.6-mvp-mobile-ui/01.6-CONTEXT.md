# Phase 1.6: MVP Mobile UI - Context

**Gathered:** 2026-06-11
**Status:** Ready for planning

<domain>
## Phase Boundary

Redesign the entire UI to match global design standards (shadcn/ui + Tailwind + lucide-react + dark mode), restructure the homepage to be the recordings list with a "New Transcript" CTA, and add export functionality to the single-recording view. Also adds three date fields per recording to support "Recorded" and "Transcribed" metadata display.

</domain>

<decisions>
## Implementation Decisions

### Homepage Structure
- **D-01:** `/` becomes the recordings list page (replaces the current splash/landing page). The existing `/recordings/` page is removed or redirects to `/`.
- **D-02:** The upload button is labeled **"New Transcript"** (not "Upload" or "New Recording") and lives in the sticky header, right-aligned.
- **D-03:** A **sticky header** is present on all pages: app name ("OpenOtter") on the left (links to `/`), "New Transcript" button on the right.
- **D-04:** Empty state: centered "No transcripts yet" message + prominent "New Transcript" button below. Clean, minimal.

### Design System
- **D-05:** Install **shadcn/ui** properly (full init + component installs). The existing hand-rolled Tailwind components are rewritten to use shadcn primitives.
- **D-06:** Install `lucide-react` for icons, `tailwindcss-animate` for shadcn transitions, `next-themes` for dark mode.
- **D-07:** Match global CLAUDE.md standards: skeleton loading states (shadcn `Skeleton`), `Suspense` boundaries, `loading.tsx` on every route, sonner for toasts.
- **D-08:** Mobile-first layout: `max-w-2xl mx-auto` container, no horizontal scroll, readable on iPhone Safari.

### Recording Dates
- **D-09:** Store **three date fields** in the SQLite `recordings` table:
  - `file_created_at` — captured from `file.lastModified` in the browser at upload time (the file's original date from the device)
  - `recorded_at` — extracted from audio file metadata (m4a/mp3 creation date tag) at upload time using `music-metadata` npm package; nullable
  - `transcribed_at` — timestamp when transcription status changes to `completed`
- **D-10:** Display logic on cards and detail pages:
  - "Recorded: [date]" → use `recorded_at` if available, else fall back to `file_created_at` with a brief note "(file date)"
  - "Transcribed: [date]" → use `transcribed_at`
- **D-11:** `file.lastModified` is sent from the client as part of the upload request body alongside the file.

### Single Recording Page (Detail)
- **D-12:** All available metadata displayed at the top: title (filename stripped of extension), Recorded date, Transcribed date, Duration, Speaker count, Confidence score (if available).
- **D-13:** **Export button** on the detail page:
  - If the full content (metadata + transcript) is ≤ ~50KB when serialized: copy to clipboard, show "Copied to clipboard" toast.
  - If larger: download as `.md` file (markdown format with metadata header + speaker-labeled transcript).
  - Format is always Markdown (not plain .txt) — speaker labels and timestamps render cleanly as markdown.
- **D-14:** Export filename derived from the recording title (filename without extension) + date, e.g. `meeting-2026-06-11.md`.

### Claude's Discretion
- Exact color palette and component variants — follow shadcn defaults.
- Whether to show confidence score as a percentage or omit if below a threshold.
- Loading skeleton shape for recording cards.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `RecordingCard`: Has the right shape (link → title + metadata). Rewrite with shadcn Card component.
- `SyncedTranscript`: Core audio player + synced transcript logic is solid — keep the logic, update visual styling.
- `formatDuration` and `formatDate` helpers in RecordingCard — keep, move to a shared util.
- SQLite store at `src/lib/db.ts` — Add column migration for three new date fields.

### Established Patterns
- Pages use `async` server components that `fetch` from internal API routes (`/api/recordings`).
- `max-w-2xl mx-auto px-4` container pattern used consistently — keep it.
- No client-side state management; data fetched server-side per page load.

### Integration Points
- `/api/upload` route: must receive `file.lastModified` from client, pass to db save.
- `/api/transcribe` route: must set `transcribed_at` when status changes to `completed`.
- `AudioUploader` client component: must read `file.lastModified` and include in upload request.
- SQLite schema: `recordings` table needs `recorded_at TEXT`, `file_created_at TEXT`, `transcribed_at TEXT` columns added via migration.

</code_context>
