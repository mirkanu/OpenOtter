## Summary

This phase rewrites the OpenOtter UI to use shadcn/ui design primitives, restructures the homepage as a recordings list, adds a sticky header with a "New Transcript" CTA, adds an export button to the detail page, and adds three date columns to the SQLite `recordings` table. The codebase is well-understood: five source files need rewriting and three new components need creation. The existing API routes require minor updates to thread `file_created_at` and `transcribed_at` through.

The primary technical risks are (1) shadcn/ui `init` overwriting `globals.css` and creating `tailwind.config.ts` in a project that currently has neither (which is actually fine — the project needs both for shadcn CSS variables to work), and (2) iPhone Safari's clipboard API requiring synchronous invocation from a user gesture, not from a resolved Promise. Both are well-documented and have clear solutions.

**Primary recommendation:** Install shadcn/ui via `npx shadcn@latest init` (which will create `tailwind.config.ts`, `postcss.config.mjs`, and rewrite `globals.css` with CSS variable tokens), then add components individually. The existing `globals.css` is minimal (three custom properties) and safe to discard after manually porting any needed rules.

### Pattern 1: shadcn/ui Init on Existing Project

**What:** `npx shadcn@latest init` creates `tailwind.config.ts`, `postcss.config.mjs`, rewrites `globals.css` with CSS variable tokens, and creates `src/lib/utils.ts` with the `cn()` helper (clsx + tailwind-merge). It also creates `components.json`.

**What gets overwritten:**
- `globals.css` — WILL be overwritten. The current file has only 3 custom properties and a body gradient. Back it up; the gradient can be removed (shadcn uses flat backgrounds).
- `tailwind.config.ts` — WILL be created (currently does not exist). This is needed for shadcn CSS variables and `tailwindcss-animate` plugin.
- `postcss.config.mjs` — WILL be created (currently does not exist). Required for Tailwind v3 to process CSS.

**What is NOT touched:** All source files in `src/app/` and `src/components/`. Only config-layer files.

**Steps:**
```bash
# 1. Commit or back up globals.css first
git stash   # or just read it — it's 28 lines

# 2. Run init — choose:
#    Style: Default
#    Base color: Neutral
#    CSS variables: Yes (required for dark mode)
npx shadcn@latest init

# 3. Install components
npx shadcn@latest add card button badge skeleton sonner
```

[CITED: https://github.com/shadcn-ui/ui/issues/2030 — confirmed overwrites globals.css and tailwind.config]

### Pattern 2: SQLite Column Migration (better-sqlite3)

**What:** Add columns to an existing table with live data using `ALTER TABLE ADD COLUMN`. SQLite's `ALTER TABLE ADD COLUMN` is safe and O(1) — it adds the column definition without rewriting table data.

**Constraint:** New columns must be nullable (no NOT NULL without a DEFAULT) because existing rows have no value.

**Pattern — idempotent migration in `database.ts`:**
```typescript
// Source: SQLite ALTER TABLE documentation
function runMigrations(db: Database.Database): void {
  const columns = db.pragma('table_info(recordings)') as Array<{name: string}>;
  const columnNames = columns.map(c => c.name);

  if (!columnNames.includes('file_created_at')) {
    db.exec('ALTER TABLE recordings ADD COLUMN file_created_at TEXT');
  }
  if (!columnNames.includes('recorded_at')) {
    db.exec('ALTER TABLE recordings ADD COLUMN recorded_at TEXT');
  }
  if (!columnNames.includes('transcribed_at')) {
    db.exec('ALTER TABLE recordings ADD COLUMN transcribed_at TEXT');
  }
}
```

Call `runMigrations(db)` inside `getDatabase()` after the `CREATE TABLE IF NOT EXISTS` block. [CITED: https://sqlite.org/lang_altertable.html]

### Pattern 4: music-metadata — Extract recorded_at from Audio Buffer

**What:** Parse audio file buffer (available in the `/api/upload` route after reading the file) to extract the `common.date` field, which contains the creation date tag in M4A/MP3 files.

**API:**
```typescript
import { parseBuffer } from 'music-metadata';
// or for CJS version:
const { parseBuffer } = require('music-metadata');

// In /api/upload route, after reading file into buffer:
const arrayBuffer = await file.arrayBuffer();
const buffer = Buffer.from(arrayBuffer);

let recordedAt: string | null = null;
try {
  const metadata = await parseBuffer(buffer, { mimeType: file.type });
  // common.date is "YYYY-MM-DD" string if present
  // For iPhone Voice Memos: may be in common.date or common.year only
  recordedAt = metadata.common.date ?? null;
} catch {
  // metadata extraction failure should not block upload
  recordedAt = null;
}
```

**Date field availability in M4A (iPhone Voice Memos):**
- `common.date` — string "YYYY-MM-DD" — most likely to contain creation date [CITED: https://github.com/borewit/music-metadata/blob/master/doc/common_metadata.md]
- `common.year` — number (year only) — fallback
- `common.originaldate` — string "YYYY-MM-DD" — for re-encoded files

**Important:** iPhone Voice Memos (`.m4a`) may not embed a `common.date` tag — the app records the date but may store it in an Apple-specific atom not exposed by `common.date`. In that case `recorded_at` will be null and the display will fall back to `file_created_at` per D-10. [ASSUMED — needs runtime verification with actual Voice Memo file]

### Pattern 5: Export — Clipboard API on iOS Safari

**The problem:** `navigator.clipboard.writeText()` on iOS Safari MUST be called synchronously from within a user gesture handler. Calling it from an async context (inside an `await`-resolved Promise) fails with "NotAllowedError: The request is not allowed by the user agent or the platform in the current context." [CITED: https://wolfgangrittner.dev/how-to-use-clipboard-api-in-safari/]

**The fix:** Pass a Promise to `navigator.clipboard.write()` — the Clipboard API holds the user gesture context open while the Promise resolves.

```typescript
// ExportButton.tsx — Safari-compatible clipboard write
async function exportToClipboard(content: string) {
  try {
    // Option A: navigator.clipboard.writeText — works if called synchronously
    // On iOS, wrap in ClipboardItem to keep gesture context alive
    await navigator.clipboard.write([
      new ClipboardItem({
        "text/plain": Promise.resolve(new Blob([content], { type: "text/plain" })),
      }),
    ]);
    toast.success("Copied to clipboard");
  } catch {
    // Fallback: ClipboardItem may not be available in older Safari
    try {
      await navigator.clipboard.writeText(content);
      toast.success("Copied to clipboard");
    } catch {
      // Ultimate fallback: download instead
      downloadAsFile(content, filename);
      toast.success("Downloaded as .md");
    }
  }
}
```

**Simpler approach that works in practice:** Per the CONTEXT.md, the 50KB threshold switches from clipboard to download. For most transcripts (< 50KB), use `navigator.clipboard.writeText(content)` called from the click handler directly — this IS synchronous from user gesture. The issue only arises when the content is computed asynchronously before calling writeText. Since the content can be computed synchronously from already-loaded data on the detail page, the simple approach works:

```typescript
// onClick handler — content is built synchronously, then writeText is called
// This preserves the user gesture context
const handleExport = async () => {
  const content = buildMarkdownExport(recording); // synchronous
  const bytes = new TextEncoder().encode(content).length;
  if (bytes <= 50 * 1024) {
    await navigator.clipboard.writeText(content); // called immediately
    toast.success("Copied to clipboard");
  } else {
    downloadMarkdownFile(content, filename);
    toast.success("Downloaded as .md");
  }
};
```

### Pattern 6: File Download on iOS Safari

**The situation:** The `<a download>` attribute is ignored by iOS Safari for cross-origin URLs and inline navigation. For Blob URLs created client-side, the behavior improved in iOS 13+ but has had regressions (notably iOS 17.4.1 had a bug, fixed in 17.5). [CITED: https://developer.apple.com/forums/thread/751063]

**Reliable approach for text/markdown files:**
```typescript
function downloadMarkdownFile(content: string, filename: string) {
  const blob = new Blob([content], { type: "text/markdown;charset=utf-8" });
  const url = URL.createObjectURL(blob);
  const a = document.createElement("a");
  a.href = url;
  a.download = filename;
  document.body.appendChild(a);
  a.click();
  document.body.removeChild(a);
  URL.revokeObjectURL(url);
}
```

This works on modern iOS Safari (iOS 14+). For the rare case where it opens inline instead of downloading, the user can use "Share → Save to Files" — acceptable for a personal tool. [ASSUMED — behavior may vary by iOS version]

### Anti-Patterns to Avoid

- **Calling `useTheme()` without a mounted guard:** Causes hydration mismatch because theme is `undefined` on server. Always guard with `useState(false)` + `useEffect`.
- **Calling clipboard API after an async operation completes:** Breaks on iOS Safari. Build the content synchronously before calling `writeText`.
- **Using `ALTER TABLE` with NOT NULL and no DEFAULT:** Fails on SQLite when existing rows have no value. All three new date columns must be nullable TEXT.
- **Assuming `common.date` is always populated:** iPhone Voice Memos may not set this tag. Always provide null fallback.
- **Running `npx shadcn@latest init` without committing first:** It will overwrite `globals.css`. Commit or note the existing content first.

## Common Pitfalls

### Pitfall 1: shadcn init Overwriting globals.css
**What goes wrong:** Developer runs `npx shadcn@latest init` and loses hand-rolled CSS rules from the existing `globals.css`.
**Why it happens:** shadcn init always rewrites globals.css to inject its CSS variable block.
**How to avoid:** Read current `globals.css` before running init. The existing file has only a body gradient background and 3 custom properties — none of which should be preserved (shadcn's CSS variables replace them). Safe to discard after verifying no custom rules are lost.
**Warning signs:** If the gradient background disappears after init — that is expected and correct.

### Pitfall 2: Tailwind Config Not Present
**What goes wrong:** `npx shadcn@latest init` may prompt about missing `tailwind.config.ts`. The project currently has Tailwind 3 installed but no config file (Next.js 15.3+ can auto-detect Tailwind without a config in some modes).
**Why it happens:** shadcn/ui requires a `tailwind.config.ts` to add the `tailwindcss-animate` plugin and configure content paths for the `@shadcn` component paths.
**How to avoid:** Let shadcn init create the config. It will set up content paths covering `src/**/*.{ts,tsx}`, which is correct. No manual editing needed.
**Warning signs:** If shadcn init completes but components have no styling — check if `tailwind.config.ts` was created with the `plugins: [require("tailwindcss-animate")]` line.

### Pitfall 3: next-themes Hydration Warning on html Element
**What goes wrong:** Console shows `Warning: Extra attributes from the server: class` on the `<html>` element.
**Why it happens:** next-themes adds a `class` attribute to `<html>` client-side (e.g., `dark`) that doesn't exist in the SSR output.
**How to avoid:** Add `suppressHydrationWarning` to the `<html>` tag in `layout.tsx`. This suppresses the warning for one level of children, which is exactly what's needed.
**Warning signs:** Hydration warning specifically on `<html>` (not on child elements).

### Pitfall 4: music-metadata ESM in Next.js API Route
**What goes wrong:** `import { parseBuffer } from 'music-metadata'` (v11 ESM) may cause bundling errors in Next.js API routes depending on the `serverExternalPackages` config.
**Why it happens:** Next.js bundles server-side code by default; pure ESM packages may not be handled correctly without marking them as external.
**How to avoid:** Use `music-metadata@cjs` (v7.14.0) which is CommonJS. If using v11+, add to `next.config.js`:
```js
const nextConfig = {
  serverExternalPackages: ['music-metadata'],
};
```
**Warning signs:** Build error mentioning "Cannot use import statement in a module" or similar ESM/CJS conflict.

### Pitfall 5: iOS Safari Clipboard API Failure
**What goes wrong:** Export button shows no feedback and clipboard is not populated on iPhone.
**Why it happens:** `navigator.clipboard.writeText()` called from an async context after the gesture fires is rejected by WebKit's permission model.
**How to avoid:** Ensure the markdown content is built synchronously before the clipboard call. Since recording data is already loaded in the detail page, this is straightforward.
**Warning signs:** Silent failure (no toast) or "NotAllowedError" in console on device.

### Pitfall 6: Missing `file_created_at` Transmission
**What goes wrong:** `file_created_at` is NULL for all recordings even though `file.lastModified` is available in the browser.
**Why it happens:** `file.lastModified` must be explicitly read from the File object in `AudioUploader.tsx` and included in the upload request. The current code does not do this.
**How to avoid:** In `AudioUploader.tsx`, before `formData.append("file", file)`, add `formData.append("fileLastModified", file.lastModified.toString())`. In `/api/upload`, read `formData.get("fileLastModified")` and convert to ISO string for the DB write.
**Warning signs:** `file_created_at` column is always NULL after migration.

```typescript
// Source: https://github.com/borewit/music-metadata README
import { parseBuffer } from 'music-metadata'; // CJS v7 or ESM v11+

async function extractRecordedAt(buffer: Buffer, mimeType: string): Promise<string | null> {
  try {
    const metadata = await parseBuffer(buffer, { mimeType });
    return metadata.common.date ?? null; // "YYYY-MM-DD" or undefined
  } catch {
    return null; // never block upload on metadata failure
  }
}
```

```typescript
function buildMarkdownExport(recording: RecordingMetadata): string {
  const lines: string[] = [
    `# ${recording.filename.replace(/\.[^.]+$/, '')}`,
    '',
    `**Recorded:** ${formatDate(recording.recorded_at ?? recording.file_created_at ?? '')}`,
    recording.transcribed_at ? `**Transcribed:** ${formatDate(recording.transcribed_at)}` : '',
    recording.audio_duration ? `**Duration:** ${formatDuration(recording.audio_duration)}` : '',
    recording.speaker_count ? `**Speakers:** ${recording.speaker_count}` : '',
    '',
    '---',
    '',
  ];
  for (const utterance of recording.utterances ?? []) {
    lines.push(`**${utterance.speaker}:** ${utterance.text}`);
    lines.push('');
  }
  return lines.filter((l, i) => !(l === '' && lines[i - 1] === '')).join('\n');
}
```
