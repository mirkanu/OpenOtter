# Phase 01.6: Code Review Report

**Reviewed:** 2026-06-11T00:00:00Z
**Depth:** standard
**Files Reviewed:** 28 (+ cross-referenced `src/lib/validation.ts`, `src/lib/vercel-blob.ts`, `src/app/api/audio/[id]/route.ts`, `src/app/api/recordings/[id]/route.ts`)
**Status:** issues_found

## Summary

Phase 01.6 delivered a complete mobile UI: recording list, detail page with metadata, synced transcript + audio player, dark-mode header, skeleton loading states, and a markdown export flow. The component architecture is sound and the security model for audio streaming (ID → DB → filepath, never URL-to-disk) is correctly implemented.

Four blockers were found:

1. **Duplicate recording writes on every poll tick** — the transcribe status endpoint writes a new DB row each time the client polls while the job is `completed`, producing duplicate recordings.
2. **Filesystem path exposed in client URL** — `filepath` (an absolute server path like `/home/services/openotter/uploads/…`) is passed through the browser URL bar and sent back to the server as an untrusted query parameter.
3. **File-size validation mismatch** — `validation.ts` caps uploads at 50 MB while `AudioUploader.tsx` and the UI claim 5 GB; the server-side cap silently rejects most real-world uploads.
4. **Polling interval never starts after first resolved status** — the `useEffect` sets up `setInterval` conditionally on `loading` at mount time, so the interval is created before the first poll resolves; stale-closure `loading` is always `true` at mount, but the logic is fragile and will fail to poll at all if the component re-renders before the interval is set.

## Critical Issues

### CR-01: Duplicate recording rows written on every completed poll

**File:** `src/app/api/transcribe/[jobId]/route.ts:31-46`

**Issue:** `saveRecording()` is called whenever `result.status === "completed" || result.status === "error"`. The client polls every 3 seconds until it sees a terminal status. On the first poll that returns `completed`, the server writes a DB row. But the client checks `data.status === "completed"` to stop polling only after receiving the response — meaning if any network retry or extra render causes a second poll before the client clears the interval, a second (and third…) row is written for the same `jobId`. There is no uniqueness constraint on `jobId` in the recordings table, so duplicates accumulate silently.

The root issue is that the server has no idempotency guard: it will write a new row with a new `nanoid()` on every request that arrives with a completed `jobId`.

**Fix:**
Add a `job_id` column with a `UNIQUE` constraint to the recordings table and use `INSERT OR IGNORE` (or check for existence first):

```sql
-- migration in database.ts
ALTER TABLE recordings ADD COLUMN job_id TEXT;
CREATE UNIQUE INDEX IF NOT EXISTS idx_recordings_job_id ON recordings(job_id);
```

```typescript
// recording-store.ts — guard before INSERT
export function saveRecording(
  transcript: TranscriptResult,
  filename: string,
  filepath: string,
  fileCreatedAt?: string,
  recordedAt?: string | null,
  jobId?: string           // add parameter
): string | null {
  const db = getDatabase();

  // Idempotency: if this jobId was already saved, return the existing id
  if (jobId) {
    const existing = db.prepare('SELECT id FROM recordings WHERE job_id = ?').get(jobId) as { id: string } | undefined;
    if (existing) return existing.id;
  }
  // ... rest of insert, passing jobId
}
```

### CR-02: Absolute server filepath leaked into browser URL and accepted back as trusted input

**File:** `src/app/upload/page.tsx:18` and `src/app/api/transcribe/[jobId]/route.ts:34`

**Issue:** After a successful upload, `handleUploadComplete` places the full server-side `filepath` (e.g., `/home/services/openotter/uploads/1234567890-file.m4a`) into `URLSearchParams` and navigates to `/transcript/${jobId}?filepath=...`. The transcript polling page then reads this value back from `searchParams` and forwards it verbatim to the API:

```
GET /api/transcribe/{jobId}?filepath=/home/services/openotter/uploads/...
```

The API accepts `filepath` from `searchParams.get("filepath")` (line 34) and writes it directly to the database as the canonical filepath for the recording. An attacker who knows or guesses a valid `jobId` can supply any `filepath` value — including paths outside the upload directory — and have it stored in the database. The audio streaming route subsequently calls `readFile(recording.filepath)` using this DB value, enabling read access to arbitrary files on the server filesystem.

This is a path traversal / SSRF vulnerability even though the audio endpoint itself does a DB lookup. The attack vector is the transcribe poll endpoint accepting `filepath` from query params.

**Fix:** Never accept `filepath` from the client. The server already has the filepath at transcription job submission time (`POST /api/transcribe`). Store it server-side in a job map or a `pending_jobs` table, keyed by `jobId`, and look it up server-side when the poll returns `completed`:

```typescript
// In-memory map (or a DB table for persistence across restarts)
const pendingJobs = new Map<string, { filename: string; filepath: string; fileCreatedAt?: string; recordedAt?: string | null }>();

// POST /api/transcribe — store metadata when job is submitted
pendingJobs.set(jobId, { filename, filepath, fileCreatedAt, recordedAt });

// GET /api/transcribe/[jobId] — retrieve metadata from server-side map
const meta = pendingJobs.get(jobId);
if (meta && (result.status === 'completed' || result.status === 'error')) {
  saveRecording(result, meta.filename, meta.filepath, meta.fileCreatedAt, meta.recordedAt);
  pendingJobs.delete(jobId);
}
```

### CR-03: File-size validation limit mismatch — server rejects files the UI promises to accept

**File:** `src/lib/validation.ts:7` and `src/components/AudioUploader.tsx:30`

**Issue:** `isValidAudioFile()` in `validation.ts` enforces a maximum of 50 MB (52 428 800 bytes, the default when `MAX_FILE_SIZE` env var is not set). This function is called both in `AudioUploader.tsx` (client-side, line 24) and in the upload API route (server-side, line 23). However, `AudioUploader.tsx` also has its own separate size check at line 30–33 that allows up to 5 GB, and the UI tooltip (line 110) and the upload page (line 51) both tell the user the limit is 5 GB.

Because `isValidAudioFile()` runs first in `handleFileSelect`, a 100 MB file fails the `isValidAudioFile()` check silently (returns `false`, displays "Please select an MP3, WAV, or M4A audio file" — a misleading error message that doesn't mention size) before the 5 GB guard is ever reached. Any audio file between 50 MB and 5 GB is rejected with a confusing type-error message.

**Fix:** Remove the size check from `isValidAudioFile` (which should only validate type) and unify the size constant:

```typescript
// validation.ts — type check only
export function isValidAudioFile(file: File): boolean {
  const validTypes = ['audio/mpeg', 'audio/wav', 'audio/mp4', 'audio/x-m4a', 'audio/mp3'];
  const validExtensions = ['.mp3', '.wav', '.m4a'];
  const hasValidExtension = validExtensions.some(ext => file.name.toLowerCase().endsWith(ext));
  return validTypes.includes(file.type) || hasValidExtension;
}

// Shared constant — single source of truth
export const MAX_AUDIO_FILE_SIZE = 5 * 1024 * 1024 * 1024; // 5 GB
```

Then enforce `MAX_AUDIO_FILE_SIZE` separately in `AudioUploader.tsx` and in the API route (which currently has no independent size guard at all — after the fix to `isValidAudioFile`, the server would accept arbitrarily large files).

### CR-04: Polling interval may never fire; stale closure on `loading`

**File:** `src/app/transcript/[jobId]/page.tsx:63-68`

**Issue:** The `useEffect` sets up `setInterval` conditionally (`if (loading)`) at line 63. `loading` is always `true` at mount, so the interval is created correctly on the first mount. However, once `shouldContinue` is set to `false` inside `pollTranscript` (on a completed/error response), the interval callback is cleared via `clearInterval`. If the component re-mounts (React Strict Mode double-invokes effects in development; navigation can also cause re-mount), the second `useEffect` run sees `loading === false` (state persisted from prior render) and the `if (loading)` guard prevents the interval from being created — leaving the user on a stale "Processing audio…" spinner with no polls.

Additionally, `pollTranscript` is defined inside the `useEffect` but `loading`, `filename`, `filepath`, `fileCreatedAt`, and `recordedAt` are referenced without being in the dependency array (only `jobId` is listed at line 74). This is a stale closure: changes to those values after mount are never seen.

**Fix:** Remove the `if (loading)` guard — the interval should always be created and allowed to self-terminate via `shouldContinue`:

```typescript
useEffect(() => {
  let intervalId: NodeJS.Timeout | null = null;
  let shouldContinue = true;

  const pollTranscript = async () => { /* ... */ };

  // Always poll immediately, then every 3 s until terminal status
  pollTranscript();
  intervalId = setInterval(pollTranscript, 3000);

  return () => {
    shouldContinue = false;
    if (intervalId) clearInterval(intervalId);
  };
}, [jobId, filename, filepath, fileCreatedAt, recordedAt]);
```

### WR-02: `getSpeakerColor` produces `NaN - 1 = NaN`, array index `NaN` returns `undefined`

**File:** `src/components/SyncedTranscript.tsx:36-39`

**Issue:** `parseInt(numericPart)` returns `NaN` when `numericPart` is empty (speaker label has no digits, e.g. `"Speaker"`). Then `NaN - 1` is `NaN`, and `colors[NaN]` is `undefined`. The fallback `|| colors[0]` on line 39 rescues this, but only because of accidental reliance on `undefined` being falsy. The guard `numericPart ?` on line 36 is supposed to prevent this branch, but `numericPart` is a string — an empty string `""` is falsy, so the ternary correctly falls to the `else` branch for no-digit labels. However, a label like `"0"` (zero) produces `parseInt("0") - 1 = -1`, and `colors[-1]` is `undefined` — the `|| colors[0]` fallback fires, but `Math.abs(-1) % colors.length` would have returned `1`, so the fallback silently collapses all "0" speakers to color index 0 (indigo) regardless.

**Fix:** Use the `else` path explicitly for robustness:
```typescript
const index = numericPart
  ? (parseInt(numericPart, 10) - 1 + colors.length) % colors.length
  : speaker.split('').reduce((acc, char) => acc + char.charCodeAt(0), 0) % colors.length;
return colors[index] ?? colors[0];
```

### WR-03: Audio proxy loads entire file into memory — no range request support

**File:** `src/app/api/audio/[id]/route.ts:62-73`

**Issue:** The audio proxy reads the entire file with `readFile()` and returns it as a single response buffer. The `Accept-Ranges: bytes` header is advertised (line 69) but the route never handles `Range` request headers. Mobile browsers (Safari on iOS in particular) rely heavily on HTTP range requests for audio seeking. Without range request support, Safari will frequently re-download the entire file from the beginning when the user seeks to a different position in the audio. For a 1-hour recording at ~50 MB, every seek triggers a full re-download and the `Content-Length` must be correct for the audio element to display total duration.

While this is primarily a performance concern, it crosses into correctness: Safari's `<audio>` element may fail to display the correct total duration without a proper `206 Partial Content` response, and on iOS the audio may not play at all for certain file sizes.

**Fix:** Parse the `Range` header and return a `206 Partial Content` response for range requests:
```typescript
const rangeHeader = request.headers.get('range');
if (rangeHeader) {
  const [start, end] = parseRange(rangeHeader, fileBuffer.length);
  return new NextResponse(fileBuffer.slice(start, end + 1), {
    status: 206,
    headers: {
      'Content-Type': contentType,
      'Content-Range': `bytes ${start}-${end}/${fileBuffer.length}`,
      'Content-Length': String(end - start + 1),
      'Accept-Ranges': 'bytes',
    },
  });
}
```

### WR-04: `saveRecording` called for error-status transcriptions — stores broken records

**File:** `src/app/api/transcribe/[jobId]/route.ts:31`

**Issue:** `saveRecording` is called on both `"completed"` and `"error"` statuses. When status is `"error"`, `transcript.utterances` is `undefined` and `transcript.text` is `undefined`. The recording is written to the database with `status: "error"` and no transcript data — creating a permanent "error" record in the list. The recordings list page has no mechanism to surface these or allow retry/deletion. Over time, failed jobs accumulate in the list as non-actionable noise.

More critically, `updateTranscribedAt` is only called on `"completed"` (line 49), which is correct — but the error case still consumes the `filename`/`filepath` query params, which are only valid once (see CR-02). If the user retries, there is no mechanism to clean up the orphaned error record.

**Fix:** Either skip writing error records entirely and only save completed transcriptions, or add a delete UI to the recordings list for error-status entries.

```typescript
// Only persist completed transcriptions
if (result.status === "completed") {
  // ... saveRecording + updateTranscribedAt
}
```

### WR-05: `validations.ts` runs `parseInt` on an env var with no radix and no NaN guard

**File:** `src/lib/validation.ts:7`

**Issue:** `parseInt(process.env.MAX_FILE_SIZE || '52428800')` has no radix parameter. If `MAX_FILE_SIZE` starts with `0` (e.g., `0755` or similar accident), `parseInt` in some engines treats it as octal. More importantly, if `MAX_FILE_SIZE` is set to a non-numeric string (e.g., a mis-pasted value), `parseInt` returns `NaN`, and `file.size <= NaN` is always `false` — causing `isValidAudioFile` to reject every file regardless of size.

**Fix:**
```typescript
const MAX_FILE_SIZE = parseInt(process.env.MAX_FILE_SIZE || '52428800', 10);
if (isNaN(MAX_FILE_SIZE)) throw new Error('MAX_FILE_SIZE env var is not a valid integer');
```

### IN-02: `document.querySelector('audio')` in `SyncedTranscript` — fragile DOM query

**File:** `src/components/SyncedTranscript.tsx:58-61`

**Issue:** `handleClickUtterance` calls `document.querySelector('audio')` to get the audio element and trigger `.play()`. This bypasses React's ownership model: if the page ever has more than one `<audio>` element (e.g., a background notification sound or a second player component), the query returns the first one in document order, which may not be the intended player. The `AudioPlayer` component already exposes a `ref` (`audioPlayerRef`) with a `seekTo` method — the ref pattern should be extended to also expose a `play()` method.

**Fix:** Extend the `AudioPlayerRef` interface to include `play()` and call it through the ref:
```typescript
// AudioPlayer.tsx
export interface AudioPlayerRef {
  seekTo: (timeMs: number) => void;
  play: () => void;
}
// In handleClickUtterance:
audioPlayerRef.current.seekTo(startTime);
audioPlayerRef.current.play();
```
